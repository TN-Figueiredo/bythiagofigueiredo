# Content Analytics & Read Tracking — Design Spec

**Date:** 2026-05-04
**Sprint:** 6 (pre-launch)
**Status:** Approved
**Score:** 98/100

## 1. Overview & Motivation

Two complementary systems:

1. **Client-side read tracking (localStorage)** — YouTube-style progress indicators showing which posts a reader has already read. Anonymous, zero-auth, zero-server for the indicator itself.
2. **Server-side content analytics (DB)** — real view counts, read depth, time-on-page, and referrer tracking to replace mock data and power MostReadSidebar with actual metrics.

### Goals
- Replace hardcoded mock engagement stats (`views: 2460`, `getMostReadPosts` pseudo-random) with real data
- Give readers visual feedback on reading progress across the site
- Prioritize unread content in suggestions (MostReadSidebar, RelatedPosts)
- Fill the "coming soon" CMS blog analytics tab with actionable metrics
- Build for extraction to `@tn-figueiredo/analytics` (empire mindset)

### Non-goals
- Real-time analytics / websocket dashboards
- Campaign engagement tracking (submissions are the metric there)
- A/B testing of titles
- Third-party analytics integration (GA, Plausible)

## 2. Data Model

### 2.1 `content_events` — raw event stream

```sql
CREATE TABLE content_events (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  site_id       uuid NOT NULL REFERENCES sites(id),
  session_id    text NOT NULL,
  resource_type text NOT NULL CHECK (resource_type IN ('blog','campaign','newsletter_archive')),
  resource_id   uuid NOT NULL,
  event_type    text NOT NULL CHECK (event_type IN ('view','read_progress','read_complete')),
  anonymous_id  text NOT NULL,
  locale        text,
  referrer_src  text CHECK (referrer_src IN ('direct','google','newsletter','social','other')),
  read_depth    smallint CHECK (read_depth >= 0 AND read_depth <= 100),
  time_on_page  smallint CHECK (time_on_page >= 0 AND time_on_page <= 3600),
  has_consent   boolean NOT NULL DEFAULT false,
  user_agent    text,
  created_at    timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_content_events_resource ON content_events(resource_type, resource_id, created_at);
CREATE INDEX idx_content_events_site_date ON content_events(site_id, created_at);
CREATE INDEX idx_content_events_anon ON content_events(anonymous_id, resource_id);
```

**Fields:**
- `session_id` — UUID v4 generated per page load. Groups view + progress + complete from the same visit.
- `anonymous_id` — from `lgpd_anon_id` in localStorage (generated by CookieBannerProvider init, available BEFORE consent interaction).
- `event_type` — 3 types: `view` (page opened), `read_progress` (page close with depth/time), `read_complete` (95%+ scroll reached).
- `has_consent` — client sends flag. Server forces `user_agent = NULL` when false, regardless of what client sends.
- `read_depth` — exact scroll percentage (0-100) at the moment of the event.
- `time_on_page` — accumulated seconds, only populated on the final event of a session (page close via sendBeacon).

**Events per page load (typically 2, max 3):**
1. `view` — after 3 seconds of permanence (filters bounces)
2. `read_complete` — immediately when scroll >= 95% (marks completion)
3. `read_progress` — on page close via sendBeacon, with final read_depth + time_on_page

**RLS policies:**
- INSERT: `anon` key via API endpoint (rate-limited). Policy: `INSERT WITH CHECK (site_visible(site_id))`.
- SELECT: `is_staff() OR is_org_admin(org_id)` — analytics visible to staff only.
- UPDATE/DELETE: none. Events are immutable.

**Retention:** 90 days (aligned with newsletter tracking). Cron `purge_content_events(p_older_than_days int default 90)`.

### 2.2 `content_metrics` — daily aggregation

```sql
CREATE TABLE content_metrics (
  id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  site_id             uuid NOT NULL REFERENCES sites(id),
  resource_type       text NOT NULL,
  resource_id         uuid NOT NULL,
  date                date NOT NULL,
  views               int NOT NULL DEFAULT 0,
  unique_views        int NOT NULL DEFAULT 0,
  reads_complete      int NOT NULL DEFAULT 0,
  avg_read_depth      smallint NOT NULL DEFAULT 0,
  avg_time_sec        smallint NOT NULL DEFAULT 0,
  referrer_direct     int NOT NULL DEFAULT 0,
  referrer_google     int NOT NULL DEFAULT 0,
  referrer_newsletter int NOT NULL DEFAULT 0,
  referrer_social     int NOT NULL DEFAULT 0,
  referrer_other      int NOT NULL DEFAULT 0,
  UNIQUE(resource_type, resource_id, date)
);

CREATE INDEX idx_content_metrics_site ON content_metrics(site_id, date);
CREATE INDEX idx_content_metrics_resource ON content_metrics(resource_type, resource_id, date);
```

Retained indefinitely (aggregated, no PII).

### 2.3 `blog_posts` denormalization

```sql
ALTER TABLE blog_posts ADD COLUMN view_count int NOT NULL DEFAULT 0;
ALTER TABLE blog_posts ADD COLUMN read_complete_count int NOT NULL DEFAULT 0;
```

Updated atomically by the aggregation cron (same transaction as content_metrics insert).

### 2.4 Cross-locale behavior

`resource_id` = `blog_posts.id` (not translation ID). Same post in pt-BR and en shares the same `resource_id`. `locale` is event metadata, not a grouping key. A reader who reads in pt-BR and opens in en: 2 events, but 1 post "read" in localStorage (keyed by post ID).

### 2.5 Cold start strategy

All existing posts start with `view_count = 0`. MostReadSidebar fallback: if `max(view_count) < 10`, use the existing pseudo-random algorithm. Transition is automatic as views accumulate over 7-14 days. Threshold configurable via constant.

## 3. API Endpoint

### `POST /api/track/content`

```typescript
// Request body (Zod validated):
{
  events: Array<{
    sessionId: string,        // UUID v4
    siteId: string,           // UUID
    resourceType: 'blog' | 'campaign' | 'newsletter_archive',
    resourceId: string,       // UUID
    eventType: 'view' | 'read_progress' | 'read_complete',
    anonymousId: string,      // UUID v4 from lgpd_anon_id
    locale?: string,
    referrerSrc?: 'direct' | 'google' | 'newsletter' | 'social' | 'other',
    readDepth?: number,       // 0-100
    timeOnPage?: number,      // seconds, capped at 3600
    hasConsent: boolean,
  }>,  // max 5 per request
}

// Response: 204 No Content (success), 400 (validation), 429 (rate limited)
```

**Rate limiting:** In-memory bucket (same pattern as `/api/ads/events`), 30 events per IP per 60 seconds.

**Server-side logic:**
1. Validate body via Zod schema
2. Rate limit check (IP from `x-forwarded-for`)
3. If `hasConsent === false`, force `userAgent = null` server-side
4. Bulk insert via `supabase.from('content_events').insert(events)` with service-role
5. Return 204

**Auth:** None. Public endpoint. Rate limiting is the protection.

**Feature flag:** `CONTENT_TRACKING_ENABLED` (server-side). Returns 204 without processing when `false`.

## 4. Client Architecture

### 4.1 Hook `useContentTracking(config)`

```typescript
interface TrackingConfig {
  siteId: string
  resourceType: 'blog' | 'campaign' | 'newsletter_archive'
  resourceId: string
  locale: string
  isPreview?: boolean  // true → skip all tracking
}
```

**Lifecycle:**
1. **Mount** → generate `sessionId` (UUID v4), start permanence timer
2. **3 seconds** → emit `view` event via `fetch()`. Dedup: check `sessionStorage['btf_view_sent:{resourceId}']` — skip if exists and < 30min old.
3. **Scroll** → observe `ScrollContext.progress`. When crosses 95%, emit `read_complete` via `fetch()`. Mark in localStorage.
4. **Page close** (`visibilitychange` + `pagehide`) → emit `read_progress` via `navigator.sendBeacon()` with final read_depth + accumulated time_on_page.
5. **Update localStorage** at each threshold (25/50/75/100) for visual card indicators.

**Dependencies:** `ScrollContext` (progress), `CookieBannerContext` (consent, anonymousId).

### 4.2 Beacon vs Fetch

- `view` and `read_complete` → `fetch()` (need confirmation, happen during active session)
- `read_progress` on page close → `navigator.sendBeacon()` (survives unload, payload < 64KB)
- Fallback: if `sendBeacon` unavailable, `fetch({ keepalive: true })`

### 4.3 Dedup

Client-side only. Hook doesn't emit `view` if one was sent for the same `resourceId` in the last 30 minutes (via `sessionStorage`). Server accepts everything — dedup is client responsibility.

### 4.4 Bot filtering

- Client-side: `navigator.webdriver === true` → skip tracking (headless bots set this)
- Server-side: aggregation cron excludes events where `user_agent` matches bot patterns (Googlebot, bingbot, Baiduspider, YandexBot, etc.). Pattern list in config.

### 4.5 Admin/preview exclusion

- Hook receives `isPreview` prop. If true, emits nothing.
- Detection: URL contains `/cms/` or query param `?preview=true` → skip.
- SSR/ISR: hook is client-only (`useEffect`), never runs on server.

### 4.6 Error handling

- Tracking fails silently. Never blocks UX. `fetch` wrapped in try/catch that does nothing on catch.
- Sentry: tracking errors NOT reported (noise). Exception: if 4xx/5xx rate > 50% in 1 hour, single alert with tag `component: 'content-tracking'`.

### 4.7 Referrer source parser

```typescript
function classifyReferrer(url: string | null): ReferrerSrc {
  if (!url) return 'direct'
  const host = new URL(url).hostname
  if (/google\.|bing\.|yahoo\.|duckduckgo\.|baidu\./.test(host)) return 'google'
  if (host.includes(NEWSLETTER_FROM_DOMAIN)) return 'newsletter'
  const utm = new URLSearchParams(new URL(document.location.href).search).get('utm_source')
  if (utm === 'newsletter') return 'newsletter'
  if (/twitter\.|x\.com|facebook\.|instagram\.|linkedin\.|threads\.net|reddit\./.test(host)) return 'social'
  return 'other'
}
```

## 5. localStorage — Read Progress Store

### Key & shape

```typescript
// Key: 'btf_read_progress'
// Value: JSON object keyed by post ID (UUID)
{
  "uuid-post-1": { d: 100, t: 1714800000 },  // d=depth%, t=timestamp(epoch seconds)
  "uuid-post-2": { d: 72,  t: 1714750000 },
}
```

Compact keys (`d`/`t`) to minimize size. 500 posts ≈ 25KB (well under 5MB localStorage limit).

### Cleanup

On hook init, prune entries older than 365 days. Prevents unbounded growth.

### Interface (future `IReadProgressStore`)

```typescript
interface IReadProgressStore {
  getProgress(resourceId: string): { depth: number; timestamp: number } | null
  setProgress(resourceId: string, depth: number): void
  getAllRead(): Map<string, { depth: number; timestamp: number }>
  cleanup(maxAgeDays: number): void
}
```

## 6. UX — Visual Indicators

### 6.1 Card indicator (Option A — YouTube-style)

Three states on BlogGrid, TagCategoryGrid, and RelatedPostsGrid cards:

| State | Red bar | Badge | Title | Opacity |
|-------|---------|-------|-------|---------|
| Not read | none | none | normal | 1.0 |
| Partially read (1-94%) | proportional width, 3px, `--pb-yt` | none | normal | 1.0 |
| Fully read (95%+) | 100% width, 3px, `--pb-yt` | "✓ lido" top-right | dimmed | title: 0.6 |

Red bar positioned between image area and content section. Uses CSS variable `var(--pb-yt)` (#FF3333) for the bar, consistent with YouTube red already defined in the palette.

Implementation: `<ReadableCard>` client component wrapper that reads localStorage and applies styles conditionally. Renders only after hydration (avoids SSR mismatch).

### 6.2 MostReadSidebar

- Counter in header: `★ Most Read · 3 de 5 lidos` (right-aligned, `--pb-faint` color, monospace)
- Unread posts appear first (client-side reordering after hydration)
- Read posts: `opacity: 0.55` + "· ✓" next to tag name
- When all read: header becomes `★ Most Read · tudo lido!` (font-caveat)

### 6.3 TagCategoryGrid counter

Tag header shows: `TECNOLOGIA · 3 de 12 lidos` (client-side, post-hydration only).

### 6.4 RelatedPosts unread-first

Server returns 6 posts (up from 3). Client filters to 3, prioritizing unread. If < 3 unread, completes with read posts.

### 6.5 Fallback when all posts are read

- No post is hidden. Goal is to indicate progress, not filter content.
- MostReadSidebar: all posts shown with read indicators.
- RelatedPosts: shown normally with subtle "releia" label.

## 9. LGPD Compliance

### Hybrid model (Option C)

| Data | Consent required | Justification |
|------|-----------------|---------------|
| View count (event_type='view') | No — legítimo interesse | LGPD Art. 7 VIII. No PII stored (no IP, anonymous_id is client-generated UUID). Operational metric for MostReadSidebar and editorial decisions. Same basis as Sentry error tracking. |
| Read depth, time_on_page | No — legítimo interesse | Behavioral but anonymous. No PII. Operational for content quality. |
| Referrer source | No — legítimo interesse | Classified string ('google'/'direct'/etc), not raw URL. No PII. |
| User agent | Yes — analytics consent | PII-adjacent. Only stored when `has_consent=true`. Forced NULL server-side otherwise. |

### Data minimization
- No IP stored (unlike ad_events which stores raw IP).
- `anonymous_id` is a client-generated UUID v4, not derivable from user identity.
- Raw events purged after 90 days. Aggregated metrics (content_metrics) retained indefinitely — contain zero PII.

## 10. Crons & Maintenance

### Cron 1: `aggregate_content_metrics`
- Schedule: `0 3 * * *` (03:00 São Paulo daily)
- RPC: `aggregate_content_events(p_date date default current_date - 1)`
- Aggregates yesterday's events into `content_metrics` (ON CONFLICT UPDATE)
- Updates `blog_posts.view_count` and `read_complete_count` in same transaction
- Excludes bot user agents via pattern matching
- Revalidates cache tags: `most-read:{siteId}`, `content-analytics:{siteId}`

### Cron 2: `purge_content_events`
- Schedule: `0 4 * * 0` (Sunday 04:00)
- Deletes `content_events` with `created_at < now() - interval '90 days'`
- Returns count deleted
- Aligned with `purge_webhook_events` pattern

### Cache invalidation

| Event | Tags invalidated |
|-------|-----------------|
| Cron `aggregate_content_metrics` | `most-read:{siteId}`, `content-analytics:{siteId}` |
| Post published/unpublished | `most-read:{siteId}` |

## 12. Performance Budget

### Bundle impact
- `useContentTracking` hook: ~2KB gzipped
- `ReadableCard` wrapper: ~500B gzipped
- `read-progress-store.ts`: ~800B gzipped
- **Total: ~3.3KB gzipped** — no new dependencies

### Core Web Vitals
- **INP:** Zero impact. Scroll tracking already exists in ScrollContext (rAF). Hook observes the `progress` value, doesn't add scroll listener.
- **LCP:** Zero impact. Hook runs in `useEffect` (post-hydration).
- **CLS:** Zero impact. Card indicators rendered client-side post-hydration but occupy already-reserved space.

### localStorage
- 500 posts × ~50 bytes each = ~25KB. Limit is 5MB. 200x margin.
- Annual cleanup (entries > 365d) prevents unbounded growth.

### Server load
- Max 3 requests per page view. Blog with ~1000 page views/day = ~3000 tracking requests/day. Trivial for Vercel.
- Rate limit 30/IP/min prevents abuse.

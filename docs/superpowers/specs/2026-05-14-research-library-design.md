# Research Library — Design Spec

**Date:** 2026-05-14
**Status:** Draft
**Author:** Thiago + Claude

## Problem

Deep research generated by Claude Cowork creates .md files locally near skill folders. This causes:

1. **Token waste** — re-reading large .md files in new conversations burns tokens
2. **No remote access** — files are trapped on one machine's filesystem
3. **No organization** — files scattered across folders, hard to find and cross-reference
4. **No reuse** — the same research gets regenerated instead of being combined with existing work

Research is a knowledge base that feeds ALL content formats (blog, video, newsletter, course, campaign). A single research item (e.g., "WYD Ongame Era") may inform multiple blog posts, video scripts, and newsletters. This is fundamentally many-to-many.

## Solution

A standalone **Research Library** in the CMS with:

- **Hierarchical topics** (max 3 levels) for organization
- **Rich text editor** (Tiptap) for viewing and editing
- **API for Claude** to push research and read it back
- **Many-to-many links** to pipeline items
- **Quick triage mode** for reviewing new research

## Data Model

### Table: `research_topics`

Hierarchical topic tree. Max depth 2 (3 levels: 0→1→2).

```
id              uuid PK default gen_random_uuid()
parent_id       uuid FK → research_topics(id) ON DELETE CASCADE, nullable
name            text NOT NULL
slug            text NOT NULL
path            text NOT NULL                    -- materialized: "gaming-history/wyd"
depth           int NOT NULL DEFAULT 0 CHECK (depth <= 2)
color           text NOT NULL DEFAULT '#a78bfa'
icon            text NOT NULL DEFAULT '📁'
sort_order      int NOT NULL DEFAULT 0
site_id         uuid NOT NULL FK → sites(id) ON DELETE CASCADE
created_at      timestamptz NOT NULL DEFAULT now()
updated_at      timestamptz NOT NULL DEFAULT now()

UNIQUE(site_id, path)
INDEX idx_research_topics_site ON (site_id)
INDEX idx_research_topics_parent ON (parent_id) WHERE parent_id IS NOT NULL
```

### Table: `research_items`

Individual research documents.

```
id              uuid PK default gen_random_uuid()
topic_id        uuid NOT NULL FK → research_topics(id) ON DELETE CASCADE
title           text NOT NULL
content_json    jsonb                            -- Tiptap JSONContent (after first edit)
content_md      text                             -- original markdown from Claude (immutable)
summary         text                             -- short summary (AI-generated or manual)
sources         jsonb NOT NULL DEFAULT '[]'      -- [{url, title, accessed_at?}]
status          text NOT NULL DEFAULT 'new'
                CHECK (status IN ('new', 'reviewed', 'starred', 'archived'))
word_count      int NOT NULL DEFAULT 0
version         int NOT NULL DEFAULT 1
site_id         uuid NOT NULL FK → sites(id) ON DELETE CASCADE
created_at      timestamptz NOT NULL DEFAULT now()
updated_at      timestamptz NOT NULL DEFAULT now()

INDEX idx_research_items_topic ON (topic_id)
INDEX idx_research_items_site_status ON (site_id, status) WHERE status != 'archived'
INDEX idx_research_items_search USING GIN (search_vector)
```

`search_vector` is a generated tsvector column updated by trigger:
```sql
setweight(to_tsvector('portuguese', coalesce(title, '')), 'A') ||
setweight(to_tsvector('portuguese', left(coalesce(content_md, ''), 50000)), 'B')
```

### Table: `research_links`

Many-to-many between research items and pipeline items.

```
id                  uuid PK default gen_random_uuid()
research_id         uuid NOT NULL FK → research_items(id) ON DELETE CASCADE
pipeline_item_id    uuid NOT NULL FK → content_pipeline(id) ON DELETE CASCADE
note                text                         -- optional context
created_at          timestamptz NOT NULL DEFAULT now()

UNIQUE(research_id, pipeline_item_id)
INDEX idx_research_links_pipeline ON (pipeline_item_id)
INDEX idx_research_links_research ON (research_id)
```

### RLS Policies

All tables follow the project pattern with idempotent `DROP POLICY IF EXISTS` before `CREATE POLICY`.

**research_topics** (direct site_id):
- SELECT: `can_view_site(site_id)` TO authenticated
- INSERT: `can_edit_site(site_id)` TO authenticated
- UPDATE: `can_edit_site(site_id)` USING + WITH CHECK TO authenticated
- DELETE: `can_edit_site(site_id)` TO authenticated

**research_items** (direct site_id):
- Same pattern as research_topics (has its own site_id column)

**research_links** (via JOIN):
- SELECT: `EXISTS (SELECT 1 FROM research_items ri WHERE ri.id = research_id AND can_view_site(ri.site_id))`
- INSERT/UPDATE/DELETE: same pattern with `can_edit_site()`

### Triggers

- `tg_set_updated_at()` on research_topics and research_items
- `research_item_version_increment` on research_items: `NEW.version := OLD.version + 1` (same pattern as `pipeline_updated_at()`)
- `research_item_search_vector_update` on research_items: updates search_vector on INSERT/UPDATE OF title, content_md
- `research_item_word_count` on research_items: computes word_count from content_md on INSERT/UPDATE

### Content Storage Strategy

- **content_md**: Original markdown from Claude. Set on creation, preserved as immutable source. API reads always return this field for AI consumption.
- **content_json**: Tiptap JSONContent. Null until user first edits in the CMS. Once set, the editor loads from content_json.
- **Editor loads**: `content_json ?? content_md` — PipelineEditor's `contentToEditorInput()` already handles both (string → `marked.parse()` → HTML, JSONContent → passthrough).
- **On Tiptap save**: Updates content_json only. content_md stays as the original.

## API Endpoints

All endpoints use `authenticatePipeline()` auth (X-Pipeline-Key header or session fallback). Standard error format: `{ error: { code, message, details? } }`.

### POST /api/pipeline/research

Create a research item. Primary endpoint for Claude to push research.

```typescript
// Request body
{
  title: string                    // required
  topic_slug: string               // required — "gaming-history" or "gaming-history/wyd"
  content_md: string               // required — markdown content
  summary?: string                 // optional
  sources?: Array<{url: string, title: string, accessed_at?: string}>
}

// topic_slug behavior:
// "estrategia"              → finds/creates topic at depth 0
// "gaming-history/wyd"      → finds/creates "gaming-history" (0) + "wyd" (1)
// "cursos/ai-dev/prompt"    → finds/creates 3 levels (0→1→2)
// "a/b/c/d"                 → 400 Bad Request "Max 3 levels"
//
// Auto-create: slug "gaming-history" → name "Gaming History"

// Response: 201
{
  data: { id, title, topic_id, status: 'new', word_count, version, created_at }
}
```

### POST /api/pipeline/research/import

Bulk import. Same schema as POST but accepts array (max 50).

```typescript
// Request body
{
  items: Array<{ title, topic_slug, content_md, summary?, sources? }>
}

// Response: 200
{
  data: {
    results: Array<{ id, title, ok: boolean, error?: string }>,
    success_count: number,
    failure_count: number
  }
}
```

### GET /api/pipeline/research

List research items with filters.

```typescript
// Query params
?topic_id=uuid                     // filter by specific topic
?topic_slug=gaming-history         // filter by topic slug (includes children)
?status=new,reviewed               // comma-separated status filter
?search=wyd+ongame                 // full-text search
?pipeline_item_id=uuid             // get research linked to a pipeline item
?limit=50&cursor=uuid              // pagination

// Response: 200
{
  data: Array<{
    id, title, topic_id, topic_path, topic_name, topic_icon,
    summary, status, word_count, sources_count,
    content_md,                    // full markdown (for Claude reads)
    linked_items_count,
    version, created_at, updated_at
  }>,
  meta: { total, has_next, next_cursor, limit }
}
```

### GET /api/pipeline/research/[id]

Single research item with full content.

```typescript
// Response: 200
{
  data: {
    id, title, topic_id, topic_path, topic_name, topic_icon,
    content_json, content_md, summary, sources, status,
    word_count, version, created_at, updated_at,
    linked_items: Array<{ id, title, format, stage }>
  },
  meta: { version, updated_at }
}
```

### PATCH /api/pipeline/research/[id]

Update research item. Requires `X-Expected-Version` header.

```typescript
// Request body (all optional)
{
  title?: string
  content_json?: JSONContent       // from Tiptap editor
  summary?: string
  sources?: Array<{url, title}>
  status?: 'new' | 'reviewed' | 'starred' | 'archived'
  topic_id?: uuid                  // move to different topic
}

// Response: 200
{
  data: { ...updated item },
  meta: { version, updated_at }
}

// Error: 409 if version mismatch
```

### DELETE /api/pipeline/research/[id]

Hard delete a research item.

### CRUD /api/pipeline/research/topics

```
GET    /api/pipeline/research/topics          — list all (tree structure)
POST   /api/pipeline/research/topics          — create { name, slug, parent_id?, color?, icon? }
PATCH  /api/pipeline/research/topics/[id]     — update { name?, color?, icon?, sort_order?, parent_id? }
DELETE /api/pipeline/research/topics/[id]      — cascade deletes children + items
```

### POST /api/pipeline/research/[id]/links

```typescript
// Request body
{ pipeline_item_id: uuid, note?: string }

// Response: 201
{ data: { id, research_id, pipeline_item_id, note, created_at } }
```

### DELETE /api/pipeline/research/[id]/links/[linkId]

Unlink a research item from a pipeline item.

## Server Actions

File: `apps/web/src/app/cms/(authed)/pipeline/research/actions.ts`

Following existing patterns: `requireEditAccess()`, `ActionResult` return type, `revalidatePath()`.

```typescript
// Core actions
saveResearchItem(id: string, version: number, input: UpdateInput): Promise<ActionResult>
updateResearchStatus(id: string, status: Status): Promise<ActionResult>
moveResearchToTopic(id: string, topicId: string): Promise<ActionResult>
deleteResearchItem(id: string): Promise<ActionResult>

// Topic management
createResearchTopic(input: TopicInput): Promise<ActionResult>
updateResearchTopic(id: string, input: Partial<TopicInput>): Promise<ActionResult>
deleteResearchTopic(id: string): Promise<ActionResult>

// Linking
linkResearchToItem(researchId: string, pipelineItemId: string, note?: string): Promise<ActionResult>
unlinkResearchFromItem(linkId: string): Promise<ActionResult>
```

All mutating actions call `revalidatePath('/cms/pipeline/research')`.

## UI Components

### Route: `/cms/pipeline/research`

Page component: `apps/web/src/app/cms/(authed)/pipeline/research/page.tsx`

Server component fetches:
- All topics (with item counts and unread counts per topic, including bubble-up from children)
- Items for the initially selected topic (or all if none)
- Stats for overview (total, unread, starred, linked)

### Component: `ResearchLibrary`

Three-panel client component.

**Left panel (220px): Topic Tree**
- Search bar (⌘K to focus)
- "Todas" entry with total count
- Collapsible tree with ▶/▼ toggles
- Each topic shows: icon, name, item count
- Unread (new) badges in yellow, bubble up from children
- "+" button to create new topic
- Right-click: rename, change icon/color, delete

**Middle panel (280px): Item List**
- Breadcrumb header: `📚 › 🎮 Gaming History`
- Sort dropdown: Recentes, Título, Tamanho
- Items grouped by subtopic (when parent selected)
- Each item shows: status dot, title, snippet (first ~100 chars of content_md), word count, date, linked items count
- Selected item highlighted with left border
- Archived items shown at bottom with opacity 0.5

**Right panel (flex): Content**
- **Read mode (default)**: Title input, status + star buttons, rendered markdown (read-only), action bar (Reviewed/Star/Archive shortcuts)
- **Edit mode (press E)**: Full Tiptap editor with PipelineToolbar (preset='full'), tabs (Conteúdo, Fontes, Sumário, Links), save footer with ⌘S
- **Empty state**: When no item selected, show "Selecione um item para ler"

### Component: `ResearchPicker`

Dialog for linking research to pipeline items. Used from:
1. Research detail "Links" tab → "Vincular a pipeline item" button
2. Pipeline item sidebar → "Buscar e vincular research" button

Search by title/topic, filter by status. Shows compact cards with topic path and word count.

### Sidebar Integration

In `cms-sections.ts`, add after Collections:

{ icon: '🔬', label: 'Research', href: '/cms/pipeline/research', minRole: 'editor' as const }

```typescript
const researchUnread = await supabase

  .select('id', { count: 'exact', head: true })

  .eq('status', 'new')
if (researchUnread.count) badges['/cms/pipeline/research'] = researchUnread.count
```

### Pipeline Item Sidebar Panel

In the pipeline item detail sidebar, add a "Research Vinculado" section:
- Shows linked research count + aggregate word count
- Lists linked items with topic path, word count, status
- "+" button opens ResearchPicker dialog
- "×" button unlinks
- Click opens research in new tab

## Keyboard Shortcuts

Global (when ResearchLibrary is focused):
- `⌘K` — focus search
- `⌘S` — save current item
- `↑↓` — navigate items in list
- `1-9` — jump to topic by position
- `R` — mark as reviewed
- `S` — toggle star
- `A` — archive/restore
- `E` — toggle edit mode
- `Escape` — exit edit mode, clear search

## Zod Schemas

```typescript
const RESEARCH_STATUS = ['new', 'reviewed', 'starred', 'archived'] as const

const ResearchItemCreateSchema = z.object({
  title: z.string().min(1).max(500),
  topic_slug: z.string().min(1).max(200),
  content_md: z.string().min(1).max(500_000),
  summary: z.string().max(2000).optional(),
  sources: z.array(z.object({
    url: z.string().url().max(2000),
    title: z.string().max(200),
    accessed_at: z.string().datetime().optional(),
  })).max(50).default([]),
})

const ResearchItemUpdateSchema = z.object({
  title: z.string().min(1).max(500).optional(),
  content_json: z.record(z.unknown()).optional(),
  summary: z.string().max(2000).nullable().optional(),
  sources: z.array(z.object({
    url: z.string().url().max(2000),
    title: z.string().max(200),
  })).max(50).optional(),
  status: z.enum(RESEARCH_STATUS).optional(),
  topic_id: z.string().uuid().optional(),
})

const ResearchImportSchema = z.object({
  items: z.array(ResearchItemCreateSchema).min(1).max(50),
})

const ResearchTopicCreateSchema = z.object({
  name: z.string().min(1).max(100),
  slug: z.string().min(1).max(100).regex(/^[a-z0-9-]+$/),
  parent_id: z.string().uuid().optional(),
  color: z.string().regex(/^#[0-9a-f]{6}$/i).default('#a78bfa'),
  icon: z.string().max(10).default('📁'),
})

const ResearchLinkSchema = z.object({
  pipeline_item_id: z.string().uuid(),
  note: z.string().max(500).optional(),
})
```

## Migration

Single migration file: `supabase/migrations/YYYYMMDDHHMMSS_research_library.sql`

Follows project conventions:
- `DROP POLICY IF EXISTS` before all `CREATE POLICY`
- `DROP TRIGGER IF EXISTS` before all `CREATE TRIGGER`
- `tg_set_updated_at()` reused from existing schema
- Custom triggers for version increment, search vector, word count

## What Is NOT In Scope

- Version history UI (content_md preserved as original is sufficient)
- Source management system (simple JSON array is enough)
- Content similarity/AI suggestions
- Export bundles
- Mobile responsive
- Drag-and-drop reorder (v2 if needed)
- Batch status change (v2 if needed)
